# FastqCA

FastqCA is a FASTQ compression tool that supports both **Lossy** and **LossLess** modes, with multi-process block-based execution for large sequencing datasets.

The command-line entrypoint is `main_new.py`, which uses the bundled `lpaq8` binary as the backend entropy encoder.

---

## Features

- Compress and decompress FASTQ files.
- Two compression strategies:
  - **Lossy**: maps/models bases and quality scores for higher compression ratio.
  - **LossLess**: preserves information for scenarios requiring reversibility.
- Multi-process parallelism with block-based processing for large files.
- Optional retention of intermediate artifacts for debugging/analysis.
- Read-count manifest validation in Lossy mode to check decompression integrity.

---

## Repository Structure

- `main_new.py`: unified CLI entrypoint.
- `Lossy_thread.py`: lossy compression/decompression logic (multi-process).
- `LossLess_thread.py`: lossless compression/decompression logic (multi-process).
- `lpaq8` / `lpaq8.exe` / `lpaq8.cpp` / `lpaq8.py`: LPAQ8 encoder-related files.
- `tools.py`: utility helpers (filesystem size/time display, etc.).
- `DATASETS.md`: benchmark dataset notes and download references.

## Requirements

- Python 3.9+ (recommended)
- Linux/macOS (using `./lpaq8`) or Windows (using `lpaq8.exe`)

Python dependencies (based on imports in this repo):

- `biopython`
- `numpy`
- `pillow`
- `tqdm`

Install dependencies:

```bash
pip install biopython numpy pillow tqdm
```

> Note: `main_new.py` currently resolves `lpaq8` as `f"{os.getcwd()}/lpaq8"`, so make sure `lpaq8` exists in your current working directory and has execute permission.

## Quick Start

### 1) Lossy Compress

```bash
python main_new.py \
  --compressor Lossy \
  --mode compress \
  --input_path /path/to/sample.fastq \
  --output_path /path/to/output_dir \
  --threads 8 \
  --block_size 134217728 \
  --save False
```

### 2) Lossy Decompress

```bash
python main_new.py \
  --compressor Lossy \
  --mode decompress \
  --input_path /path/to/output_dir/sample.lossy \
  --output_path /path/to/restore_dir \
  --threads 8 \
  --save False
```

### 3) LossLess Compress

```bash
python main_new.py \
  --compressor LossLess \
  --mode compress \
  --input_path /path/to/sample.fastq \
  --output_path /path/to/output_dir \
  --threads 8 \
  --block_size 134217728 \
  --save False
```

### 4) LossLess Decompress

```bash
python main_new.py \
  --compressor LossLess \
  --mode decompress \
  --input_path /path/to/output_dir/sample.lossless \
  --output_path /path/to/restore_dir \
  --threads 8 \
  --save False
```

## CLI Arguments

| Argument | Required | Default | Description |
|---|---|---|---|
| `--compressor` | No | `Lossy` | Compressor type: `Lossy` / `LossLess` |
| `--input_path` | Yes | - | Input file path (`.fastq` for compression, archive for decompression) |
| `--output_path` | Yes | - | Output directory or output base path |
| `--mode` | Yes | - | `compress` (or `c`) / `decompress` (or `d`) |
| `--save` | No | `False` | Keep intermediate files: `True` / `False` |
| `--threads` | No | CPU count | Number of worker processes |
| `--block_size` | No | `134217728` | Block size in bytes |
| `--manifest` | No | auto-derived | Read-count manifest path (Lossy mode) |

---

## Output and Validation Notes

- In **Lossy compression**, the tool counts reads in input FASTQ and writes a manifest file (default: `<archive>.readcount`).
- In **Lossy decompression**, it reads the manifest and validates read-count consistency; mismatch raises an error.
- With `--save False`, intermediate artifacts (temporary chunks and internal outputs) are removed after processing.

## Troubleshooting

### 1) `lpaq8` not found

Check the following:

1. `lpaq8` exists in your current execution directory.
2. The file is executable (`chmod +x lpaq8` on Linux/macOS).
3. On Windows, use `lpaq8.exe` and adjust path handling if needed.

### 2) Read count mismatch after decompression

This usually indicates mismatched archive/manifest files or an incomplete decompression run. Verify:

- The `--input_path` archive and `.readcount` came from the same compression run.
- Outputs/intermediate files were not overwritten or partially deleted.
- You are not mixing archives generated by different code versions.
